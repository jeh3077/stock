성공적인 로또 번호 생성기(앱/웹) 개발을 위한 PRD(제품 요구사항 정의서, Product Requirements Document) 초안을
작성해 드립니다.

이 PRD는 모바일 앱(Android/iOS)을 기준으로 작성되었으며, 프로젝트의 규모와 목적에 따라 내용을 가감하여 사용하시면 됩니다.

📄 PRD: 로또 번호 생성기 (프로젝트명: 행운의 로또)

1. 문서 개요 (Document Overview)

  - 제품명: 행운의 로또 (가칭)
  - 작성일: 2026-04-24
  - 작성자: [이름/직책]
  - 문서 버전: v1.0
  - 제품 형태: 모바일 애플리케이션 (iOS / Android)

2. 제품 목적 및 배경 (Objective & Background)

2.1 배경

  - 매주 수백만 명의 사람들이 로또(6/45)를 구매하지만, 매번 번호를 직접 고르는 것에 피로감을 느끼거나 '자동' 옵션 외에 자신만의
    조합을 원함.
  - 기존 로또 앱들은 UI/UX가 복잡하거나 과도한 광고로 인해 사용자 경험(UX)이 떨어지는 경우가 많음.

2.2 제품 목적 (비전)

  - 사용자에게 직관적이고 깔끔한 UI를 제공하여 빠르고 쉽게 로또 번호를 생성할 수 있게 한다.
  - 단순 랜덤뿐만 아니라 사용자 맞춤형 조건(제외수, 고정수 등) 및 통계 기반의 번호 생성 기능을 제공한다.
  - 생성한 번호를 저장하고, 추첨 당일 당첨 여부를 쉽게 확인할 수 있도록 돕는다.

3. 타겟 사용자 (Target Audience)

  - Primary (주 사용자): 매주 정기적으로 로또를 구매하며, 쉽고 빠르게 번호를 추천받고 싶은 20~50대 일반인.
  - Secondary (부 사용자): 과거 당첨 통계, 출현 빈도, 미출현 번호 등 데이터를 분석하여 전략적으로 번호를 조합하고 싶은 로또
    매니아.

4. 핵심 기능 요구사항 (Functional Requirements)

기능의 중요도에 따라 MoSCoW 방법론(Must-have, Should-have, Could-have)으로 분류합니다.

4.1 Must-Have (필수 기능 - MVP)

  - 단순 랜덤 번호 생성: 145 중 중복 없는 6개의 숫자를 랜덤으로 15게임 생성.
  - 조건부 번호 생성:
      - 고정수 설정: 반드시 포함할 번호 1~5개 지정.
      - 제외수 설정: 절대 포함하지 않을 번호 다수 지정.
  - 번호 저장소 (내 번호함): 생성된 번호를 로컬(기기)에 저장. (회차별 구분)
  - 최신 당첨 번호 조회: 동행복권 API를 연동하여 매주 최신 당첨 번호 표시.

4.2 Should-Have (권장 기능)

  - QR 코드 스캔: 실물 로또 용지의 QR코드를 카메라로 스캔하여 당첨 여부 즉시 확인.
  - 저장된 번호 당첨 결과 자동 대조: '내 번호함'에 저장된 번호가 추첨 후 몇 등인지 자동으로 계산하여 표시.
  - 통계 기반 생성:
      - 최근 5주/10주 미출현 번호 위주 생성.
      - 역대 최다/최소 출현 번호 기반 생성.
      - 홀짝 비율(예: 3:3, 4:2) 및 번호 총합(예: 100~150 사이) 필터링.

4.3 Could-Have (부가 기능)

  - 푸시 알림: 매주 토요일 오후 8시 45분 추첨 완료 후 결과 알림, 금요일 저녁 로또 구매 잊지 않기 알림.
  - 판매점 찾기: 위치 기반 서비스(GPS)를 이용해 내 주변 1등/2등 당첨 판매점 및 일반 판매점 지도 표시.
  - 소셜 공유: 생성된 번호를 카카오톡 등 메신저로 친구에게 공유.

5. 비기능 요구사항 (Non-Functional Requirements)

  - 성능 (Performance):
      - 번호 생성 버튼 클릭 시 0.5초 이내에 결과가 노출되어야 함.
      - 동행복권 API 호출 실패 시, 앱이 다운되지 않고 "데이터를 불러올 수 없습니다"라는 안내 메시지를 띄워야 함.
  - 사용성 (Usability):
      - 한 손으로 조작하기 쉬운 하단 탭 바 형태의 네비게이션 적용.
      - 다크 모드(Dark Mode) 지원.
      - 광고는 사용자의 번호 생성 흐름을 방해하지 않는 위치(예: 하단 배너, 결과 창 하단)에만 배치.
  - 데이터 관리: 회원의 개인정보를 요구하지 않으며, 생성된 번호 데이터는 서버가 아닌 사용자 기기(Local DB -
    SQLite/Realm 등)에 저장하여 서버 유지비용을 최소화.

6. 화면 구성 및 사용자 흐름 (UI/UX & User Flow)

6.1 주요 화면 (GNB / 하단 탭)

1.  홈 (Home):
      - 최신 회차 당첨 번호 하이라이트.
      - 원터치 '랜덤 번호 5게임 생성' 메인 버튼.
2.  생성기 (Generator):
      - 탭 1: 조건 생성 (고정수/제외수 선택 UI).
      - 탭 2: 통계 생성 (미출현, 홀짝비율 등 옵션 선택).
3.  내 번호함 (History):
      - 회차별로 내가 저장한 번호 목록 리스트.
      - 추첨 전/후 상태 라벨 표시 (예: "결과 대기", "3등 당첨!").
4.  QR 확인 (Scanner):
      - 카메라 뷰파인더 활성화 및 인식 시 결과 모달창 팝업.

6.2 핵심 User Flow (번호 생성 및 저장)

1.  앱 실행 → [생성기] 탭 이동 2.[고정수 2개, 제외수 5개] 선택 후 '생성하기' 버튼 탭
2.  번호 6개씩 5게임(조합) 결과 화면에 노출
3.  마음에 드는 조합 체크 후 [내 번호함에 저장] 버튼 탭
4.  "저장 완료" 토스트 메시지 노출

7. 시스템 아키텍처 및 외부 연동 (System & Integration)

  - 프론트엔드 (Client): Flutter 또는 React Native (크로스 플랫폼 개발로 iOS/Android 동시 출시).
  - 백엔드 (Server):
      - 초기(MVP)에는 별도의 백엔드 서버 없이 Local DB로 구동.
      - 당첨 번호 데이터는 동행복권 공식 오픈 API를 Client에서 직접 호출하여 캐싱.
          - API 예시:
            https://www.dhlottery.co.kr/common.do?method=getLottoNumber&drwNo=회차번호
  - DB: SQLite, Hive (Flutter 적용 시) 또는 Async Storage.
  - 분석 툴: Google Analytics (Firebase) - 화면 조회수, 버튼 클릭 이벤트 트래킹.

8. 출시 일정 및 로드맵 (Roadmap)

  - Phase 1 (MVP 출시 - D+4주):
      - 단순 랜덤 / 고정,제외수 생성 기능
      - 로컬 번호 저장 기능
      - 최신 당첨 번호 조회 API 연동
  - Phase 2 (업데이트 - D+8주):
      - QR 코드 스캔 기능 추가
      - 통계 기반 번호 생성 로직 추가
  - Phase 3 (수익화 및 고도화 - D+12주):
      - 푸시 알림 기능 (Firebase Cloud Messaging)
      - 주변 당첨 판매점 지도 기능 (Kakao/Naver Map API 연동)
      - 배너 및 전면 광고(AdMob) 연동

9. 성공 지표 (KPIs)

  - Acquisition (획득): 앱 스토어 월간 다운로드 수 (목표: 월 5,000건)
  - Activation (활성화): 앱 설치 후 1회 이상 번호를 생성한 유저의 비율 (목표: 80% 이상)
  - Retention (유지): 주간 활성 사용자(WAU) 재방문율. (로또 특성상 금~토요일 접속 트래픽 분석 중요)
  - Engagement (참여): 1인당 1주 평균 번호 생성 횟수, '내 번호함' 저장 횟수.
